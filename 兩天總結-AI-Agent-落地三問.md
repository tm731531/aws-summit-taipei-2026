## 1️⃣ AI Agent 的資安要考量什麼

傳統資安管「資料」和「人」。Agent 多了一件事：**它會自己做動作**。所以除了原本的加密、權限，還多了一整層「怎麼管一個會自主行動的東西」。

### 先看資料 —— 大家都漏掉「使用中」那一態

| 資料狀態 | 一般怎麼保護 | 盲點 |
|---|---|---|
| 靜態（存著） | 加密（AES 等） | 大家都會做 |
| 傳輸（移動中） | TLS／憑證 | 大家都會做 |
| **使用中（進了記憶體）** | **常常沒保護** | **一進 RAM 就是明文，連系統管理員都看得到** |

Agent 跑推論、處理 PII、拿金鑰的那一瞬間，資料是明文。這是「機密運算」（硬體隔離的可信執行環境 TEE ＋ 遠端驗證 attestation）要補的洞——**只有沒被竄改、通過度量值驗證的程式才拿得到金鑰解密**，就算主機管理員拿到 root 也讀不到那個「小房間」裡的記憶體。

### Agent 特有的攻擊面（傳統系統沒有的）

- **Prompt Injection**：有人用對話/文件內容騙 agent 做不該做的事
- **工具濫用**：agent 能呼叫工具、API、下指令，權限給太大 = 它能亂做事
- **記憶／context 洩漏**：跨 session 的狀態若沒隔離，A 用戶的資料被 B 看到（多租戶場景尤其致命）
- **PII 外洩**：agent 把不該說的個資輸出出去

### 控制清單（可直接當 checklist）

| 控制點 | 一句話 |
|---|---|
| 威脅模型 | 先問「**要防誰、不防誰**」——講清楚邊界，比假裝防全部誠實 |
| 最小權限 | 每個 agent「**能做什麼**」要明確列出，預設拒絕 |
| 動態政策 | 每個工具呼叫都過一次 allow／deny，不是一次性授權 |
| 身分 | 分 inbound（誰能叫這個 agent）與 outbound（這個 agent 能用誰的權限） |
| Guardrails | 進出都過濾：擋 prompt injection、去識別化 PII |
| **Human-in-the-loop** | **高風險動作（下單、刪除、對外）一定留人工核准**。可以讓 AI 查，但不讓 AI 拍板 |
| 可稽核 | 每個動作留 trace：誰、什麼時候、做了什麼——事後查得到 |

> **心法**：治理屬於**平台層**，不是每個 agent 各做各的。一個 agent 時你手動管沒事；一千個 agent 時，部署、工具、存取、追蹤、guardrails 全部變成平台責任。

## 2️⃣ 怎麼讓大家起手順暢

最大的敵人不是技術難度，是**害怕**和**期待被廣告養壞**。順序要對：

### 第一步：先破除「一句話生系統」的迷思
- 現實是：**不是**一句 Prompt 就跑完整個系統——AI 不知道你公司的流程、權限、規則
- 但也不用怕：不是一句話，卻也「才一個晚上」。**這兩句話同時成立**，多數人只聽過一半，不是幻滅就是放棄

### 第二步：從自己的真實痛點開始，不是從技術開始
- 挑一個你自己每天在忍的小麻煩，用**你自己專業的語言**描述（誰輸入什麼、要顯示什麼、誰負責、要看到什麼狀態）——**全程不需要技術名詞**
- 你以為的門檻是「不會寫 code」，真正的門檻是「講不清楚自己的流程」。而後者剛好是你最懂、工程師最不懂的——**那是你的優勢，不是劣勢**

### 第三步：範圍砍小 —— 這是成敗分水嶺

> 真實對比：同一家公司、同一套工具、同樣非工程背景。
> 一個人要「**一個表單**」→ 一個晚上做完。
> 一個人要「**一套系統**」→ 三個月後連畫面都還沒看到。

### 第四步：能驗證 > 會寫 code
- 非技術者最常卡在「AI 幫我寫好了，但我要怎麼執行？」
- 用容器（Docker 這類）當「**安全的測試盒子**」——重點不是看懂程式碼，是能不能把它跑起來、看到對不對
- 然後一輪一輪修：「沒指派的請顯示黃色」就是一句很好的修正指令

### 給推動的人（組織層面）
- ⚠️ **別假設 AI 自動更快**：有 RCT 研究實測資深開發者用 AI **慢了 19%**，而所有人賽前都預測會快 20–40%。原因是 coding 只是流程一小段，不改流程，省下的時間會被別的環節吃回去
- 先**量 before**（沒有基線就證明不了價值）→ 用**一個 POC** 證明價值 → 再擴散
- 心態轉變才是核心：從「**工具的操作者**」→「**流程的設計者**」

## 3️⃣ Harness / Vibe / Spec 怎麼說明

### 先給一個心智模型：AI 工程分三層（缺一不可）

| 層 | 名稱 | 做什麼 | 白話 |
|---|---|---|---|
| 1 | **Prompt** | 設計明確指令 | 教 AI「**怎麼問**」 |
| 2 | **Context** | 動態組裝對的資訊 | 給 AI「**看對的東西**」 |
| 3 | **Harness** | 建控制系統 | 當 AI 開始自己動，決定它「**能做什麼、做錯了怎麼辦**」 |

### Harness 是什麼（最容易講不清的一層）
- 一個等式：**Agent = Model + Harness**
- 比喻：Harness = **馬具**。LLM 像一隻很強、但無法完全預測的動物，馬具是駕馭牠、決定方向的那套控制系統
- 具體是六件事：上下文注入 ／ 工具系統 ／ 執行編排（含 retry、human-in-the-loop）／ 記憶與狀態 ／ 可觀測性（trace、log）／ 驗證與恢復（測試、用 LLM 當評審、自動回滾）
- 為什麼重要：**同樣的輸入、同樣的 context，跑三次結果可能不一樣**（成功／超時／幻覺）。Prompt 和 Context 解決不了這個不確定性，Harness 才行

> 附帶一個反直覺的事實：把 20 份文件全塞給 LLM，答案落在中間段時準確度只有 **52.9%**，比完全不給文件（56.1%）**還差**。連 1M token 的最新模型都逃不掉這個「lost-in-the-middle」結構性問題——**context 不是越多越好，位置和篩選才是關鍵**。

### Vibe 還是 Spec —— 不是二選一，是光譜

| | **Vibe Coding** 🎵 | **Spec-Driven** 📋 |
|---|---|---|
| 怎麼給需求 | 自然語言，邊聊邊做 | 需求 → 設計 → 任務，三階段 |
| 可預測性 | 低 | 高 |
| 適合 | 原型、學習、探索、**收尾微調** | 生產級、團隊協作、長期維護 |
| 追溯性 | 只有對話歷史 | 需求到程式碼全鏈路 |

**怎麼選（可以直接講給人聽）**
- 你對這個領域**很懂** → Vibe 快
- 你**不熟**、或這東西要**長期維護／多人協作** → Spec（先把規格理清，避免 AI「自己預設立場」生出跟你要的差很遠的東西）
- **實務上兩個混用**：Spec 搭大骨架，最後細節切回 Vibe 微調
- 讓 Spec 變可靠的關鍵技巧：用 **EARS 格式**寫需求——`When <觸發>, the system shall <行為>`，把模糊的一句話逼成可驗收的敘述

## 貫穿兩天的一條主線

這些工具和方法，不是要把每個人變工程師，也不是在拚模型多聰明。

**兩天所有場次——不管是金融平台、機密運算、HR 小工具、開發方法論——都在講同一件事：怎麼讓 AI 可控、可信、可治理。**

> **今年的主題是治理，不是 AI。**

而對個人的收束則是那句最好記的：**「AI 不會讓所有人變工程師，但 AI 會讓懂問題的人更快做出解法。你不需要成為工程師，但你可以成為 Builder。」**
